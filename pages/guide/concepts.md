# 核心概念

开始配置之前，先把几个常见概念分清楚。New API 的大多数问题，本质上都绕不开这些词。

## 渠道

渠道是 New API 连接上游模型服务的配置项。一个渠道通常包含：

- 上游地址；
- 上游密钥；
- 支持的模型；
- 渠道类型；
- 优先级或权重；
- 分组限制；
- 启用状态。

如果某个模型请求失败，首先要看对应渠道是否可用。

## 模型

模型是客户端请求时填写的 `model` 名称。

在 New API 里，模型名可以直接对应上游模型，也可以通过模型映射转成另一个上游名称。

例如客户端请求：

```json
{
  "model": "gpt-4o-mini"
}
```

New API 可以根据渠道配置，把它转发到对应上游。

## 分组

分组用于控制不同令牌能访问哪些渠道或模型。

常见用法：

- 普通用户使用默认分组；
- 高级用户使用更高额度或更强模型分组；
- 测试渠道单独放到测试分组；
- 不同业务使用不同分组隔离。

## 令牌

令牌是客户端访问 New API 的凭据。建议为不同应用创建不同令牌，方便后续查看日志、限制额度和撤销访问。

不要把管理员账号密码交给客户端，也不要让多个无关应用共用同一个令牌。

## 余额与倍率

余额、倍率和计费设置用于估算请求消耗。不同模型可以配置不同倍率，不同用户也可以有不同额度。

如果发现消耗异常，优先检查：

- 模型倍率是否配置错误；
- 渠道倍率是否符合预期；
- 是否有客户端循环请求；
- 是否有异常令牌被滥用。

## 日志

日志是排查问题最重要的入口。遇到请求失败时，先看日志里的：

- 请求模型；
- 使用渠道；
- 返回状态；
- 错误信息；
- 消耗统计；
- 请求时间。

别一上来就改配置。先看日志，少坐牢。
