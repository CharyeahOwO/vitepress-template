# 推荐使用流程

如果是第一次部署 New API，可以按这个顺序来，不要一上来就把所有渠道都塞进去。

## 1. 先跑通服务

先完成部署，确认后台可以正常打开，数据库和 Redis 等依赖没有报错。

此时不要急着接一堆模型。先保证 New API 自己是活的。

## 2. 添加一个稳定渠道

先选一个你确定可用的上游渠道，配置最少模型，测试通过后再继续扩展。

建议先验证：

- 渠道密钥是否正确；
- 上游地址是否能访问；
- 模型名称是否匹配；
- 流式响应是否正常；
- 请求日志是否记录完整。

## 3. 创建测试令牌

不要直接拿管理员权限测试客户端。创建一个普通令牌，用它接入客户端。

这样后续出问题时，可以直接从令牌维度定位请求来源。

## 4. 接入客户端

客户端通常需要配置：

```text
Base URL: https://你的域名/v1
API Key: New API 生成的令牌
Model: 后台已支持的模型名
```

如果客户端支持 OpenAI 兼容接口，一般都可以按这个方式接入。

## 5. 观察日志

测试几轮请求后，回到后台看日志。

重点确认：

- 是否走到了预期渠道；
- 是否出现重试或失败；
- token 消耗是否合理；
- 流式输出是否完整；
- 错误信息是否可读。

## 6. 再扩展多渠道

单渠道稳定后，再添加备用渠道、分组和模型映射。

不要一次性改太多。否则出了问题，你很难判断是哪一层炸了。

## 建议原则

- 先少后多；
- 先通后优；
- 先看日志再改配置；
- 每个客户端单独令牌；
- 重要渠道不要和测试渠道混在一起。
