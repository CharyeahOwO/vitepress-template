# 日志与请求排查

日志是 New API 排障的第一入口。不要靠感觉猜，先看日志。

## 看什么

一次失败请求，至少看这些信息：

- 请求时间；
- 使用令牌；
- 请求模型；
- 命中渠道；
- HTTP 状态码；
- 上游错误信息；
- token 消耗；
- 请求耗时。

## 判断问题在哪一层

可以按链路拆：

```text
客户端
  ↓
反向代理 / CDN
  ↓
New API
  ↓
渠道配置
  ↓
上游模型服务
```

如果 New API 没有日志，说明请求可能没到 New API，优先查客户端 URL、DNS、反代和防火墙。

如果 New API 有日志但上游失败，优先查渠道、Key、模型名和上游状态。

## 常见错误方向

### 请求没到 New API

表现：后台没有任何日志。

可能是：

- 域名解析错误；
- 反代路径不对；
- 客户端 Base URL 填错；
- CDN 或 WAF 拦截；
- HTTPS 证书或端口问题。

### 请求到了但渠道失败

表现：后台有日志，错误来自上游。

可能是：

- 上游 Key 失效；
- 上游模型名错误；
- 上游限流；
- 上游余额不足；
- 上游接口协议不兼容。

### 请求成功但客户端异常

表现：后台显示成功，但客户端报错。

可能是：

- 客户端不兼容返回字段；
- 流式格式解析失败；
- reasoning、tool call、多模态字段不兼容；
- 客户端对模型名称有额外限制。

## 排查建议

一次只改一个地方。

先用最简单请求验证通路，再逐步加上长上下文、流式、工具调用、多模态等复杂能力。

复杂请求失败，不代表整个渠道不可用；可能只是某个能力没有映射好。
