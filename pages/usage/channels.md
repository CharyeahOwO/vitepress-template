# 渠道配置

渠道决定 New API 如何访问上游模型服务。渠道配置错了，客户端再怎么改也没用。

## 添加渠道

添加渠道时通常需要填写：

- 渠道类型；
- 上游地址；
- 上游密钥；
- 支持模型；
- 分组；
- 优先级或权重；
- 是否启用。

配置完成后，先用后台测试功能或简单客户端请求验证。

## 模型映射

模型映射用于把客户端请求的模型名转换成上游实际模型名。

适合这些情况：

- 客户端只认固定模型名；
- 上游模型名太长；
- 多个上游想统一成同一个模型入口；
- 需要做兼容命名。

## 优先级与备用渠道

如果多个渠道支持同一个模型，可以通过优先级、权重或分组策略决定请求走向。

建议：

- 稳定渠道放前面；
- 测试渠道单独分组；
- 不要把不稳定上游混进主分组；
- 重要模型至少准备一个备用渠道。

## 渠道不可用时怎么查

优先看日志里的错误，而不是盲目改配置。

常见原因：

- 上游 Key 失效；
- 上游余额不足；
- 模型名不存在；
- 上游限流；
- 网络无法访问；
- 协议格式不兼容；
- 分组权限不匹配。

## 不要一次改太多

排查渠道问题时，一次只改一个变量。比如只改模型名、只改地址、只换 Key。

一次改一堆，最后能跑了也不知道是哪一步修好的；跑不起来更不知道哪里炸了。
